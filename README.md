## What Each Part Does

### Part 1 — clean_data.py

- Removes duplicate customers (keeps most recent `signup_date`)
- Validates and lowercases emails, adds `is_valid_email` flag
- Parses dates in 3 formats: `YYYY-MM-DD`, `DD/MM/YYYY`, `MM-DD-YYYY`
- Fills missing `region` with `"Unknown"`
- Fills missing `amount` with product-group median
- Normalizes `status` to controlled vocabulary: `completed`, `pending`, `cancelled`, `refunded`
- Drops rows where both `customer_id` and `order_id` are null
- Adds `order_year_month` derived column
- Prints full cleaning report to stdout

### Part 2 — analyze.py

- Left-joins orders → customers → products (explicit `on=` and `how=` args)
- Reports unmatched rows after each join
- Generates 5 output CSVs:
  - Monthly revenue trend (completed orders only)
  - Top 10 customers by total spend + 90-day churn flag
  - Category performance (revenue, avg order value, order count)
  - Regional analysis (customers, orders, revenue, avg per customer)

### Part 3 — Backend + Frontend

- FastAPI backend with CORS enabled, 404 error handling on missing files
- React dashboard with Recharts — line chart, bar chart, sortable table, region cards
- Live API data fetching with loading and error states
- Responsive layout (desktop 1280px + mobile 375px)
- Working buttons: Export CSV, date range filter, search, sort, churn filter, sidebar navigation

## Assumptions Made

1. **Date formats** — Three formats supported: `YYYY-MM-DD`, `DD/MM/YYYY`, `MM-DD-YYYY`. Tried in this order; first match wins.
2. **Missing amount** — Filled with median of the same product group. Falls back to global median if all values for a product are null.
3. **Status normalization** — Variants like `done`, `canceled`, `waiting` are mapped. Anything outside the controlled vocabulary defaults to `pending`.
4. **Churn definition** — No completed order in the last 90 days relative to the maximum order date in the dataset (not today), ensuring reproducible results.
5. **Duplicate customers** — Most recent `signup_date` is kept; ties broken by original row order.
6. **Regional analysis** — Order count and revenue use all orders (not just completed) to match standard business reporting conventions.
7. **Products not in orders** — Retained in `products.csv` as per the dataset description; unmatched products show as null after the left join and are reported.
